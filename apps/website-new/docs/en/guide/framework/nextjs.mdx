# Next.js

:::::danger Project Deprecation
Support for Next.js is ending [read more](https://github.com/module-federation/core/issues/3153)
:::::

`@module-federation/nextjs-mf` is now a rspack-first Next adapter with an explicit public surface:

- `@module-federation/nextjs-mf`: the Next+rspack federation plugin
- `@module-federation/nextjs-mf/server`: Pages Router SSR helpers

The package no longer exposes wildcard deep imports, `./utils`, or the legacy `extraOptions` surface.

## Scope

- Next `^12 || ^13 || ^14 || ^15 || ^16`
- Pages Router SSR
- Rspack-based builds

Wave 1 does **not** include App Router support.

## Install

Use the rspack-based Next toolchain instead of webpack-local environment hacks:

```bash
pnpm add next-rspack @rspack/core @module-federation/nextjs-mf
```

There is no `NEXT_PRIVATE_LOCAL_WEBPACK` requirement in the new adapter.

## Plugin Usage

The root entry accepts standard Module Federation options only.

```js
const withRspack = require('next-rspack');
const NextFederationPlugin = require('@module-federation/nextjs-mf');

/** @type {import('next').NextConfig} */
const nextConfig = {
  webpack(config, options) {
    config.plugins.push(
      new NextFederationPlugin({
        name: 'home',
        filename: 'static/chunks/remoteEntry.js',
        remotes: {
          shop: `shop@http://localhost:3001/_next/static/${
            options.isServer ? 'ssr' : 'chunks'
          }/remoteEntry.js`,
        },
        exposes: {
          './menu': './components/menu',
          './pages/index': './pages/index',
        },
      }),
    );

    return config;
  },
};

module.exports = withRspack(nextConfig);
```

Use explicit page exposes for any stitched Pages routes. Do not rely on `pages-map`, wildcard exports, or `exposePages`.

## Server Entry

The `./server` entry is intentionally small:

- `withFederatedRequest()`
- `flushChunks()`
- `FlushedChunks`

Use it from `pages/_document.js` so SSR asset collection stays request-scoped and manifest-backed:

```jsx
import Document, { Html, Head, Main, NextScript } from 'next/document';
import {
  FlushedChunks,
  flushChunks,
  withFederatedRequest,
} from '@module-federation/nextjs-mf/server';

class MyDocument extends Document {
  static async getInitialProps(ctx) {
    const { initialProps, chunks } = await withFederatedRequest(async () => {
      const initialProps = await Document.getInitialProps(ctx);
      const chunks = await flushChunks();

      return { initialProps, chunks };
    });

    return {
      ...initialProps,
      chunks,
    };
  }

  render() {
    return (
      <Html>
        <Head>
          <FlushedChunks chunks={this.props.chunks} />
        </Head>
        <body>
          <Main />
          <NextScript />
        </body>
      </Html>
    );
  }
}

export default MyDocument;
```

## Refresh Model

Refresh is handled inside `nextjs-mf` itself:

- remotes are resolved from `mf-manifest.json`
- request start captures the active remote generation
- manifest `buildVersion` changes activate a new generation for later requests
- in-flight requests stay pinned to the generation they started with
- old generations retire after their request refcounts drain

`revalidate()` is **not** exported from `@module-federation/nextjs-mf/server`.

## Asset Resolution

Remote assets continue to resolve against the remote origin, but the old public flags:

- `enableImageLoaderFix`
- `enableUrlLoaderFix`

are no longer part of the supported API.

## Runtime Plugins

Custom federation runtime plugins are still supported:

```js
new NextFederationPlugin({
  runtimePlugins: [require.resolve('./runtime-plugin.js')],
});
```

## Importing Remotes

```js
import React, { lazy } from 'react';

const Menu = lazy(() => import('shop/menu'));
```

Prefer `React.lazy` over `next/dynamic` for federated lazy boundaries in the Pages Router examples.
